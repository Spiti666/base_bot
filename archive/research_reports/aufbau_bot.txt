AUFBAU LIVE-BOT + BACKTESTER (ANONYMISIERT)
Datei: aufbau_bot.txt
Stand: 2026-04-02

WICHTIGER HINWEIS
- Diese Dokumentation beschreibt den kompletten technischen Aufbau, die Ablauflogik, Variablenfamilien, UI-Basis und die verwendeten Techniken.
- Es werden absichtlich keine konkreten Coin-Listen und keine konkreten Strategie-Namen aufgefuehrt.
- Alles ist auf Architektur-, Pipeline-, Risiko-, UI- und Runtime-Ebene erklaert.

=====================================================================
1) SYSTEMZIEL UND TRENNUNG
=====================================================================

Das System besteht aus zwei strikt getrennten Betriebszonen:

1. Live-Zone
- Echtzeit-Candles per WebSocket
- Signalerzeugung, Readiness, Risiko-Regeln
- Paper-Execution (simulierte Ausfuehrung)
- Laufende UI-Anzeige fuer Monitoring

2. Backtest-/Optimizer-Zone
- Historische Daten aus lokaler DB
- Simulations- und Optimierungslaeufe
- Batch-Orchestrierung ueber Coin x Strategie x Intervall
- Kompakte Ergebnisreports und Ergebnis-Persistenz

Architekturprinzip:
- Gemeinsame technische Basis (Datenmodell, Engine-Kern, UI-Rahmen)
- Saubere Trennung von Live-Flow und Backtest-Flow
- Reproduzierbarkeit von Ergebnissen als Leitprinzip

=====================================================================
5) LIVE-ABLAUF END-TO-END
=====================================================================

5.1 Start
- GUI startet Live-Thread
- Runtime-Settings werden geladen
- Historienfenster fuer Warmup wird sichergestellt

5.2 WebSocket
- Pro Symbol/Intervall werden Kanaele abonniert
- Jedes Tick/Kline-Update aktualisiert die letzte Candle
- Beim Wechsel auf neue Candle wird die alte Candle als geschlossen verarbeitet

5.3 Signal + Gate
- Signal aus aktiver Coin-Konfiguration
- Optional Setup-Gate:
  - Trend-/Lagecheck
  - Volumencheck
  - Volatilitaetscheck
  - Candle-Qualitaetscheck
- Ergebnis: approved / blocked

5.4 Readiness
- Je Coin wird readiness_score berechnet
- Readiness nutzt strategieabhaengige Eingangswerte + Coin-Profile
- Statusstates: nicht bereit / vorbereitet / ready

5.5 Trade-Ausfuehrung
- Nur wenn keine Sperre und kein Konflikt mit Positionslimits
- Positionsgroesse aus equity * risk_per_trade_pct * leverage
- Entry-Fee direkt eingebucht

5.6 Positionspflege
- Pro Candle:
  - Hard-SL / TP
  - Chandelier / Strategy-Exit
  - Trailing-Stop-Stufen
  - Breakeven-Umstellung
- Exit schreibt PnL und Fees zuruueck

5.7 Live-Status und Schutz
- Laufende Kennzahlen je Coin
- Optional Kandidatenmodus mit Schutzlogik:
  - Verlustserie
  - DD-Schwelle
  - Rolling PF
  - Fill/Slippage-Qualitaet

=====================================================================
6) BACKTEST-ABLAUF END-TO-END
=====================================================================

6.1 Trigger aus GUI
- Einzelrun oder Batchrun
- Auswahl: Coin(s), Strategie(n), Intervall(e)
- Auto-Mode kann parallel mit expliziten Strategien laufen

6.2 Queue-Bau
- Batchqueue wird aus Selektion gebaut
- Optional je Queue-Eintrag Auto-Flag
- Reihenfolge: strikt nacheinander

6.3 Historie laden
- DB-Lesezugriff, ggf. inkrementelles Sync-Fenster
- Warmup-Sicherung (mind. benoetigte Kerzen)
- Optional Date-Range-Fenster fuer gezielte Runs

6.4 Signalpfad
- Vektorisiert oder JIT
- Optional Regime-Maske
- Setup-Gate auf Candleindex-Basis

6.5 Simulationslauf
- Gleiche Engine wie im strukturierten Ausfuehrungskern
- Kostenmodell aktiv (Fee + Slippage)
- Ergebnis: Trades, PnL, DD, PF, WR, RRR, Nebenmetriken

6.6 Persistenz + Summary
- Ergebnis in DB
- Kompakte Summary-Datei
- Kachel-Update im UI

=====================================================================
7) OPTIMIZER-ABLAUF
=====================================================================

7.1 Grid-Erzeugung
- Strategie-Familien erhalten feste oder konditionale Parameterraeume
- Harte Gueltigkeitsregeln filtern ungueltige Profile

7.2 Sampling/Scan-Modi
- Full-Scan oder Sampling (konfigurierbar)
- 1-Stage oder 2-Stage (Exploration -> Verification)

7.3 Worker-Pipeline
- Multiprocessing-Pool
- Task-Payloads enthalten:
  - Profil
  - Candlepayload
  - optional precomputed signal cache key
  - Gate- und Regime-Infos

7.4 Bewertungsmetrik
- Kernmetriken: PnL, PF, WR, Trades, DD, RRR
- Zusatzhuerden:
  - Mindesttrades (intervallabhaengig)
  - Breakeven-Stabilitaetsregel (strategieabhaengig)
  - avg_profit_per_trade_net_pct

7.5 Stage-Diagnostik
- fail_min_trades
- fail_breakeven
- fail_avg_profit
- fail_multi
- eligible

7.6 Final-Verify und Konsistenz
- Bester Kandidat wird final verifiziert
- Candidate-Metriken und Final-Metriken werden verglichen
- Bei starker Abweichung: explizite Warnung

=====================================================================
13) KLARE ABLAUFKETTE (KOMPAKT)
=====================================================================

Live:
1. Config laden -> Thread starten
2. Historie/Warmup sicherstellen
3. WebSocket streamt Candles
4. Signal + Gate + Readiness
5. Entry/Exit ueber Paper-Engine
6. Persistenz + UI-Update + Monitoring

Backtest:
1. Selektion aus GUI -> Queue bauen
2. Historie laden/neiden (Date-Range + Warmup)
3. Profile evaluieren (1- oder 2-stage)
4. Bester Kandidat final verifizieren
5. Konsistenzcheck Candidate vs Final
6. Speichern + Kacheln + Summary
